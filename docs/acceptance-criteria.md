# CampusFit AI 验收标准

## 1. 文档阶段验收标准

### 1.1 完整性

以下文档必须存在且内容非占位：

1. `product-overview.md`
2. `prd-mvp.md`
3. `user-stories.md`
4. `page-tree.md`
5. `system-architecture.md`
6. `repo-structure.md`
7. `db-schema.md`
8. `api-design.md`
9. `ai-architecture.md`
10. `prompt-strategy.md`
11. `implementation-roadmap.md`
12. `acceptance-criteria.md`
13. `assumptions.md`
14. `decisions.md`
15. `progress.md`

### 1.2 一致性

1. MVP 功能范围在产品、接口、数据库文档中保持一致。
2. 页面树中的核心页面必须能在 PRD 中找到对应业务目标。
3. API 与数据库实体能建立清晰映射关系。
4. AI 文档必须体现“规则引擎负责计算，LLM 负责解释和交互”。
5. 所有用户入口描述统一为公开网页，而不是小程序或 App。

### 1.3 可执行性

1. 第二阶段可直接按文档初始化仓库。
2. 第三阶段可直接按文档实现 MVP 主链路。
3. 不确定项已沉淀到 `assumptions.md`，而不是留在隐含口径中。

## 2. 第二阶段验收标准

1. Monorepo 结构符合 `repo-structure.md`。
2. 根目录可执行 `pnpm lint`、`pnpm test`、`pnpm build`。
3. `apps/api`、`apps/web`、`apps/admin`、`services/ai-service` 可本地启动。
4. README 和 `.env.example` 完整。
5. 基础 CI 可执行代码检查。

## 3. 第三阶段 MVP 验收标准

### 3.1 登录与建档

1. 用户可完成邮箱验证码登录。
2. 首次登录用户会进入建档流程。
3. 建档完成后生成有效档案并触发初始计划生成。
4. 缺失关键字段时返回明确校验错误。

### 3.2 今日页

1. 用户进入今日页能看到当日饮食与训练摘要。
2. 当日无计划时系统可自动生成或提示重试。
3. 页面具备 loading / empty / error 状态。
4. 桌面和移动浏览器都能正常展示。

### 3.3 饮食计划

1. 饮食计划包含热量 / 宏量目标和餐次建议。
2. 至少支持食堂、宿舍、家庭三类场景之一致输出。
3. 补剂建议仅在用户开启时出现。

### 3.4 训练计划

1. 训练计划包含动作、组数、次数、休息时长和注意事项。
2. 新手与中级用户计划结构存在差异。
3. 当天无法训练时能输出休息或轻活动方案。

### 3.5 打卡

1. 用户可提交当日打卡。
2. 打卡记录会更新今日页状态。
3. 体重记录能写入历史数据。

### 3.6 周复盘

1. 用户能看到最近一周复盘结果。
2. 周复盘至少包含执行率、体重变化、问题和建议。
3. 无足够数据时要有 empty 状态说明。

### 3.7 AI 助手

1. 用户能基于今日计划发起问答。
2. AI 超时不会阻塞主业务。
3. AI 输出不修改规则计算结果。
4. 高风险问题有安全拒答。

### 3.8 商品列表

1. 用户能查看商品列表与详情。
2. 商品具备分类或标签展示。
3. 无商品数据时有 empty 状态。

## 4. 工程质量验收标准

1. 关键接口具备单元测试或集成测试。
2. 关键路径有埋点和日志。
3. 所有接口错误码统一。
4. 数据库迁移可重复执行。
5. 规则引擎输出具备版本号和追踪能力。
6. 公开网页具备基础 SEO 和浏览器兼容性校验。

## 5. 交付质量红线

出现以下情况，不视为完成：

1. 只有页面占位，没有真实接口和数据流。
2. 接口定义与数据库结构对不上。
3. AI 代替规则引擎做核心计算。
4. 页面没有 loading / empty / error 状态。
5. 仍以小程序或原生 App 作为默认主入口。
6. 决策、进度、假设文档未同步更新。
