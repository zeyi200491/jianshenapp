# CampusFit AI 假设记录

更新时间：2026-03-31

## 1. 产品假设

1. MVP 主要验证公开网页端，不要求小程序或原生 App 承载核心业务。
2. 商品模块在 MVP 阶段只提供列表与详情，不包含支付与订单闭环。
3. 周复盘按自然周或约定周起始生成，具体以实现阶段配置为准。
4. 补剂在产品中始终处于辅助角色，不作为增长主驱动。

## 2. 用户假设

1. 核心用户以 18-30 岁为主，具备基础浏览器使用能力。
2. 用户愿意输入最少量建档信息换取每日计划。
3. 用户更能接受“简单可执行建议”，而非精确称重式方案。
4. 用户可接受邮箱验证码作为 MVP 登录方式。

## 3. 业务假设

1. 校园食堂场景可通过规则模板抽象成常见菜品替代，而不做学校级精细化菜单。
2. 训练动作库在 MVP 阶段可采用内置基础动作集合，不依赖大规模视频内容。
3. 商品列表可先由运营静态维护，不依赖复杂库存系统。
4. 后台一期以运营效率为主，登录与数据管理先允许本地 mock 鉴权。

## 4. 技术假设

1. 邮箱验证码登录在本地开发阶段需要提供 mock 或测试策略，否则联调效率会过低。
2. PostgreSQL 实例支持 `pgvector` 扩展。
3. Prisma 用于事务模型管理，向量表和扩展迁移可由 SQL 补充实现。
4. AI 服务可以通过标准 HTTP 与 NestJS 解耦通信。
5. 公开网页采用响应式方案，同时兼容桌面和移动浏览器。
6. 现有 `package-lock.json` 与各处 `node_modules` 视为历史初始化残留，后续以 `pnpm` 为唯一包管理口径逐步收敛。
7. 当前开发机的 Docker Desktop 可能不稳定或不可用，因此 API 需要保留 demo 数据兜底能力，避免前端与 AI 联调被基础设施问题阻断。
8. AI 服务本地默认使用 mock provider，本轮以可运行、可联调为优先，不强制绑定正式模型供应商。
9. 当前 MVP 代码运行链路尚未实际消费 Redis，因此本机 Redis 可暂缓到缓存 / 异步任务接入阶段再落地。

## 5. 运维假设

1. 首期环境规模较小，可接受模块化单体 + 独立 AI 服务的部署模式。
2. Redis 未来既可承担缓存，也可承担部分异步任务状态存储。
3. 首期不强制引入复杂消息队列，定时任务和轻量异步可先用应用内调度或 Redis 实现。
4. 本地开发验收允许采用“真实 PostgreSQL + mock AI provider + 暂不接 Redis”的组合模式，但上线前必须收敛到真实数据库、缓存与正式模型供应商。

## 6. 待确认项

1. 周复盘的固定生成时间是每周日夜间还是用户首次访问时懒生成。
2. 商品详情是否需要跳转外部购买链接。
3. 后续是否需要补充 Google / GitHub / 微信等第三方登录，而不仅是邮箱登录。
4. 管理后台一期是否必须包含完整模板 CRUD，还是先通过数据库种子初始化。
5. 团队是否接受在开发环境中默认开启 API 自动降级模式，还是要求仅在显式 `API_DATA_MODE=mock` 时启用。
6. 若要做公开站 SEO，是否需要增加公开内容页与静态知识页。
