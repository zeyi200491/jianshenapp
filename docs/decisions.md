# CampusFit AI 决策记录

更新时间：2026-03-31

## ADR-001：MVP 主入口选择公开网页

### 状态

已确定

### 决策

MVP 以公开网页为主入口，优先打通登录、建档、今日执行、打卡、复盘和基础商品浏览。

### 原因

1. 网页可覆盖更广泛用户，不受平台限制。
2. 当前目标是验证每日执行闭环，而不是验证平台分发能力。
3. 放弃小程序和原生 App 能显著减少前期工程分散。

## ADR-002：认证方式采用邮箱验证码登录

### 状态

已确定

### 决策

MVP 采用邮箱验证码登录作为默认认证方式，不再以微信登录作为前提。

### 原因

1. 邮箱登录更适合公开网页场景。
2. 不依赖特定平台身份体系，覆盖面更广。
3. 工程接入成本和本地联调成本更低。

## ADR-003：规则引擎与 LLM 职责分离

### 状态

已确定

### 决策

规则引擎负责热量、宏量营养、训练框架和计划结构的核心计算；LLM 只负责解释、问答和复盘文案增强。

### 原因

1. 计算结果必须稳定、可测试、可追溯。
2. LLM 不适合作为核心确定性计算器。
3. 后续替换模型不应影响业务规则稳定性。

## ADR-004：后端采用 NestJS 主控 + FastAPI AI 服务

### 状态

已确定

### 决策

业务后端由 NestJS 统一编排，AI 能力由独立 FastAPI 服务承载，通过 HTTP 通信。

### 原因

1. NestJS 适合承载认证、业务模块和 API 编排。
2. Python 在模型调用、Prompt 编排和检索链路上更灵活。
3. 保持 AI 独立服务更利于演进和降级。

## ADR-005：数据库采用单 PostgreSQL 集群承载事务数据与向量数据

### 状态

已确定

### 决策

事务数据和向量检索数据共用一个 PostgreSQL 集群，启用 `pgvector` 扩展，常规模型由 Prisma 管理，向量相关迁移由 SQL 补充。

### 原因

1. 首期运维复杂度更低。
2. 数据一致性和联查更容易控制。
3. 能满足 MVP 阶段规模要求。

## ADR-006：今日页使用聚合接口

### 状态

已确定

### 决策

公开网页今日页优先走单个聚合接口返回饮食、训练、打卡状态和复盘提示，而不是前端多接口拼装。

### 原因

1. 聚合接口更适合首屏性能优化。
2. 今日页是最高频核心页面，应尽量减少前端复杂度。
3. 对桌面与移动浏览器都更稳。

## ADR-007：商品模块在 MVP 仅做基础浏览

### 状态

已确定

### 决策

商品模块在 MVP 只实现列表和详情展示，不实现购物车、支付、订单系统。

### 原因

1. 当前优先级是验证执行闭环，不是电商闭环。
2. 支付和订单会显著增加实现复杂度与合规成本。
3. 可以先验证商品曝光和点击需求。

## ADR-008：AI 服务保留在 `services/ai-service`

### 状态

已确定

### 决策

AI 服务继续位于 `services/ai-service`，不再迁移到 `apps/ai-service`。

### 原因

1. 当前代码已经按该目录落地。
2. 服务目录能更清晰表达其独立部署属性。
3. 避免文档与仓库现实再次发生冲突。

## ADR-009：不再以小程序或原生 App 作为当前阶段交付对象

### 状态

已确定

### 决策

删除小程序与 Flutter App 相关的当前阶段实现要求，所有用户侧功能统一收敛到网页端。

### 原因

1. 当前资源应集中在单一入口上。
2. 功能相同但多端并行会显著增加测试和维护成本。
3. 网页已足够验证产品价值与核心闭环。
