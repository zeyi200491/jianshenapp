# CampusFit AI MVP PRD

## 1. 文档目标

本 PRD 用于定义 CampusFit AI MVP 的业务范围、功能优先级、交付边界和验收标准，为网页产品开发提供统一约束。

## 2. MVP 目标

在公开网页端验证“每日执行闭环”是否成立，确保用户可以从登录建档开始，在同一产品内完成：

1. 生成今日饮食计划
2. 生成今日训练计划
3. 完成打卡记录
4. 查看周复盘
5. 使用基础 AI 助手问答
6. 浏览基础商品列表

## 3. 用户问题定义

### 3.1 关键问题

1. 用户不知道如何根据目标制定饮食和训练。
2. 用户缺少可执行的“今天该做什么”入口。
3. 用户没有低成本记录与反馈机制。
4. 用户对补剂和商品选择存在认知混乱。

### 3.2 MVP 解决方式

1. 用规则引擎把建档信息转为可执行计划。
2. 用今日页作为统一执行入口。
3. 用打卡和周复盘建立行为反馈。
4. 用 AI 助手解释计划，降低理解成本。
5. 用基础商品列表承接执行需求，但不做复杂推荐。

## 4. 角色与权限

### 4.1 普通用户

- 通过网页登录
- 完成建档
- 查看与执行计划
- 打卡与查看复盘
- 使用 AI 助手
- 浏览商品

### 4.2 运营人员

- 配置基础计划模板
- 维护饮食映射、训练模板与商品信息
- 查看用户执行数据

## 5. MVP 功能范围

### 5.1 登录与建档

#### 目标

建立最小可用用户画像，为规则引擎和 AI 提供输入。

#### 输入字段

1. 性别
2. 年龄
3. 身高
4. 体重
5. 目标类型：减脂 / 维持 / 增肌
6. 活动水平
7. 训练经验
8. 可训练天数
9. 饮食场景：食堂 / 宿舍简做 / 家庭烹饪
10. 饮食禁忌与偏好
11. 是否接受补剂建议

#### 输出

1. 用户基础档案
2. 初始热量和宏量目标
3. 初始训练频率与训练类型建议

### 5.2 今日页

#### 目标

作为每日执行中心，承载“看计划、去打卡、问 AI、看商品”的统一入口。

#### 页面内容

1. 今日日期与计划状态
2. 今日热量与蛋白目标
3. 今日饮食计划卡片
4. 今日训练计划卡片
5. 今日打卡入口
6. AI 快速提问入口
7. 商品入口

### 5.3 饮食计划生成

#### 目标

基于建档信息和场景生成低复杂度、可执行的饮食方案。

#### 规则要求

1. 根据目标类型生成热量目标。
2. 蛋白、碳水、脂肪按规则计算。
3. 根据饮食场景选择不同方案模板。
4. 输出以“餐次 + 食材 / 菜品建议 + 替代方案”为主，不要求精确到克级称重。

#### 输出形式

1. 早餐 / 午餐 / 晚餐 / 加餐建议
2. 每餐目标说明
3. 食堂 / 宿舍 / 家庭场景替代项
4. 补剂建议（仅在开启时展示）

### 5.4 训练计划生成

#### 目标

基于训练经验、目标和可训练天数生成今日任务。

#### 规则要求

1. 新手优先全身或上下肢分化。
2. 中级用户根据训练天数生成分化模板。
3. 训练计划必须包含动作、组数、次数、休息时长、注意事项。
4. 当天无法训练时支持休息 / 轻活动建议。

### 5.5 打卡记录

#### 目标

以最少字段记录用户执行情况，避免记录负担过重。

#### 打卡维度

1. 饮食完成度
2. 训练完成度
3. 饮水或步数等可选辅助数据
4. 体重记录
5. 主观状态：精力、饱腹感、疲劳感

### 5.6 周复盘

#### 目标

每周给出执行反馈和下周建议。

#### 输出内容

1. 本周计划执行率
2. 饮食与训练完成情况
3. 体重变化趋势
4. 行为亮点
5. 风险提醒
6. 下周建议

### 5.7 AI 问答助手

#### 目标

回答用户在执行中的常见问题，不承担核心计算职责。

#### 支持的问题类型

1. 为什么今天这样安排饮食
2. 动作怎么做
3. 某个食材没有时怎么替换
4. 今天没时间训练怎么办
5. 是否需要补剂

#### 不负责的问题

1. 医疗诊断
2. 高风险减重方案
3. 处方级营养建议

### 5.8 基础商品列表

#### 目标

提供执行相关商品浏览能力，为后续转化打基础。

#### 商品类型

1. 基础补剂
2. 简单健身器材
3. 代餐 / 轻食扩展商品

#### MVP 限制

1. 仅支持列表与详情展示。
2. 不做购物车、支付、订单闭环。

## 6. 非功能要求

1. 所有页面必须具备 loading / empty / error 状态。
2. 所有关键接口有统一错误码。
3. 所有关键行为有埋点与审计日志。
4. 规则计算必须可复现，不依赖 LLM 随机输出。
5. AI 功能必须有超时降级与兜底文案。
6. 网页端必须支持桌面浏览器与移动浏览器的响应式适配。

## 7. 业务边界

### 7.1 本期不处理

1. 复杂个性化推荐排序
2. 实时视频动作识别
3. 用户间社交和社区内容
4. 商家入驻与复杂分销
5. 原生 App 相关体验优化

### 7.2 本期允许保守处理

1. 补剂推荐以规则枚举和解释为主。
2. 商品推荐以静态标签和基础筛选为主。
3. 周复盘可先采用规则汇总 + LLM 文案增强。

## 8. MVP 核心流程

1. 用户首次进入网页并完成邮箱验证码登录。
2. 用户完成建档向导。
3. 系统自动生成今日饮食与训练计划。
4. 用户查看计划并在执行后打卡。
5. 用户可随时向 AI 助手咨询计划相关问题。
6. 每周用户查看复盘结果与下周建议。
7. 用户浏览相关商品列表。

## 9. 发布条件

满足以下条件后，才可视为开发完成：

1. 文档、数据库、接口定义一致。
2. 今日页主链路可全程运行。
3. 核心接口具备自动化测试。
4. 关键路径埋点可追踪。
5. AI 问答超时不阻塞主流程。
