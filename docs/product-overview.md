# CampusFit AI 产品总览

## 1. 产品定位

CampusFit AI 是一个面向大学生和轻居家健身人群的 AI 健身执行平台。

产品核心不是输出零散知识，而是把“饮食 + 训练 + 打卡 + 复盘 + AI 问答 + 商品浏览”串成可执行的每日闭环，让用户每天明确知道：

1. 今天该吃什么
2. 今天该练什么
3. 今天完成了多少
4. 本周坚持得怎么样
5. 当前是否需要额外补充某些基础商品

## 2. 愿景与价值主张

### 2.1 愿景

成为任何人都可以直接通过浏览器使用的 AI 健身执行入口，让健康管理从“知道很多”变成“每天做到”。

### 2.2 核心价值

1. 降低执行门槛：把复杂营养和训练计算隐藏在系统内部。
2. 提升坚持率：通过今日页、打卡、周复盘形成持续反馈。
3. 扩大可达范围：以公开网页承载核心功能，降低平台接入门槛。
4. 保持可解释性：规则引擎负责计算，LLM 负责解释与陪伴。

## 3. 目标用户

### 3.1 核心用户画像

1. 大学生健身新手：不知道如何开始，缺乏系统计划。
2. 食堂场景用户：主要在学校食堂就餐，无法精确称重和计算热量。
3. 宿舍简做用户：具备基础备餐能力，但不想自己搭配饮食。
4. 家庭饮食用户：希望兼顾日常饮食与减脂 / 增肌目标。
5. 轻中度训练用户：已经开始锻炼，但饮食与训练没有形成配套。

### 3.2 典型痛点

1. 不知道应该摄入多少热量和蛋白质。
2. 不知道今天做哪些训练动作。
3. 计划过于复杂，执行成本高。
4. 缺少持续反馈，容易中断。
5. 对补剂和器材选择认知混乱。

## 4. 产品形态

### 4.1 公开网页

定位为 MVP 的主入口，采用响应式 Web，支持桌面浏览器和移动浏览器访问，负责：

- 登录与建档
- 今日计划执行
- 饮食 / 训练查看
- 打卡记录
- 周复盘查看
- AI 助手问答
- 基础商品浏览

### 4.2 运营后台

定位为一期必要后台，负责：

- 模板管理
- 商品管理
- 内容管理
- 基础数据查看
- 运营配置维护

### 4.3 后续扩展

后续如有必要，可在网页能力稳定后再扩展 PWA 或原生 App，但不再作为当前 MVP 的前提。

## 5. 产品原则

1. 饮食优先，补剂只作补充，不反向主导用户计划。
2. 规则引擎负责计算，LLM 负责解释、引导和问答。
3. 优先解决“今天怎么执行”，再扩展更复杂能力。
4. 先做网页 MVP 主链路，再考虑平台扩展。
5. 所有实现必须可运行、可测试、可追踪。

## 6. MVP 范围定义

### 6.1 本期必须实现

1. 登录与建档
2. 今日页
3. 饮食计划生成
4. 训练计划生成
5. 打卡记录
6. 周复盘
7. 基础 AI 问答助手
8. 基础商品列表

### 6.2 本期明确不做

1. 实时动作纠错
2. 社区内容流
3. 高复杂度推荐系统
4. 大规模视频处理
5. 复杂商家系统
6. 原生 App 开发

## 7. 核心闭环

1. 用户通过网页登录并完成建档。
2. 系统根据用户目标、身体数据和偏好生成今日饮食与训练计划。
3. 用户在今日页完成查看、执行和打卡。
4. AI 助手对计划内容进行解释与陪伴式答疑。
5. 系统每周生成复盘，反馈执行情况和下一步建议。
6. 商品列表为执行提供基础补充选择，但不强行驱动购买。

## 8. 成功指标

### 8.1 MVP 核心指标

1. 建档完成率
2. 今日页次日留存率
3. 饮食计划查看率
4. 训练计划查看率
5. 打卡完成率
6. 周复盘触达率
7. AI 助手使用率
8. 商品列表点击率

### 8.2 质量指标

1. 今日页接口 P95 响应时间小于 800ms（不含 LLM）
2. 计划生成接口成功率大于 99%
3. AI 问答有超时兜底与降级话术
4. 关键行为全链路可追踪

## 9. 版本节奏建议

### V0.1 文档与基础架构

- 完成产品、架构、数据库、接口、AI 方案设计
- 初始化 Monorepo 与公共工程能力

### V0.2 网页 MVP 主链路

- 打通建档、今日页、饮食、训练、打卡、周复盘、AI 助手、商品列表

### V0.3 运营增强

- 后台内容管理、模板管理、商品管理
- 数据看板与运营配置

### V0.4 智能扩展

- 更强的知识库问答
- 更细的计划调整策略
- 更复杂的推荐和商业化能力
