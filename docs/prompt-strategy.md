# CampusFit AI Prompt 策略

## 1. 目标

为 CampusFit AI 的 AI 服务建立统一 Prompt 策略，保证输出：

1. 与规则引擎结果一致
2. 语气稳定
3. 可结构化解析
4. 具备安全边界
5. 便于在 `services/ai-service` 中模板化维护

## 2. Prompt 总原则

1. 先给角色，再给任务，再给上下文，再给输出格式。
2. 规则结果是事实来源，模型不得擅自改写核心数值。
3. 优先输出可执行建议，不输出空泛鼓励。
4. 回答应适配大学生与轻居家用户，语言简洁直接。
5. 任何补剂相关回答必须强调“饮食优先，补剂补充”。
6. 命中高风险内容时，服务层优先拦截，不让模型继续生成危险建议。

## 3. 系统 Prompt 统一约束

当前 AI 服务共享以下系统约束：

1. 你是 CampusFit AI 的执行型健身助手。
2. 你的职责是解释和引导，不是重新计算用户计划。
3. 若用户问题涉及今日计划，优先基于提供的结构化计划回答。
4. 若问题超出计划范围，可给基础原则，但不得输出医疗诊断。
5. 若涉及补剂，必须先判断是否能通过正常饮食解决。
6. 若用户提出高风险行为，必须明确劝阻并建议寻求专业帮助。
7. 输出必须为合法 JSON，不返回 Markdown 代码块。

对应模板文件：

1. `services/ai-service/app/prompts/system.txt`
2. `services/ai-service/app/prompts/diet_explanation_task.txt`
3. `services/ai-service/app/prompts/training_explanation_task.txt`
4. `services/ai-service/app/prompts/plan_adjustment_task.txt`
5. `services/ai-service/app/prompts/rag_answer_task.txt`

## 4. 场景化 Prompt 设计

### 4.1 场景一：饮食计划解释

#### 输入上下文

1. 用户目标
2. 饮食场景
3. 今日热量和宏量目标
4. 今日饮食计划摘要
5. 可选训练安排摘要
6. 用户问题
7. 检索出的饮食 / 补剂知识片段

#### 输出要求

1. 先回答问题
2. 再给 1-3 条可执行建议
3. 不改动计划核心数值
4. `riskNote` 留给补剂提醒或安全注记

#### 示例结构

```json
{
  "answer": "今天这样安排是因为你当前目标是减脂，同时你是食堂场景，计划优先保证蛋白和控制总热量。",
  "tips": [
    "午餐优先选瘦肉和两份蔬菜",
    "训练前加餐可以选牛奶或酸奶",
    "晚餐主食减半即可，不需要完全不吃"
  ],
  "riskNote": ""
}
```

### 4.2 场景二：训练计划解释

#### 输入上下文

1. 用户目标与训练水平
2. 今日训练计划摘要
3. 可选饮食安排摘要
4. 用户问题
5. 检索出的训练知识片段

#### 输出要求

1. 解释动作顺序和执行重点
2. 提供 1-3 条训练执行建议
3. 不改动计划中的组数、次数和时长目标

### 4.3 场景三：一键改计划说明

#### 目标

当用户点击固定动作按钮时，由规则引擎先完成结构化改写，再让模型输出解释。

#### 当前支持动作

1. 饮食：`canteen_no_chicken_breast`
2. 饮食：`dorm_simple_meal`
3. 训练：`no_gym_access`
4. 训练：`only_20_minutes`
5. 训练：`low_energy_day`

#### Prompt 重点

1. 明确“结构化改计划已由规则引擎完成”
2. 模型只能解释改动原因与执行方法
3. 不得二次发明新的数值变更

#### 示例输出要求

```json
{
  "answer": "已按 20 分钟场景压缩为更容易执行的版本，核心目标不变，你只需要优先完成前四个关键动作。",
  "tips": [
    "组间休息控制在 45-60 秒",
    "先做前两个主动作",
    "今天完成度优先，不追求加量"
  ],
  "riskNote": ""
}
```

### 4.4 场景四：基础 RAG 问答

#### 目标

回答食材替代、动作注意事项、补剂基础原则等常见问题。

#### 输入上下文

1. 用户问题
2. 可选用户档案
3. 可选今日计划摘要
4. 检索命中的知识片段

#### 输出要求

1. 只基于知识片段和上下文回答
2. 知识不够时明确说明适用范围有限
3. 若涉及补剂，保留“饮食优先”原则

### 4.5 场景五：高风险拒答

#### 触发条件

1. 医疗诊断与处方建议
2. 极端减脂、断食、催吐、泻药、脱水
3. 自伤、自杀倾向
4. 违禁药物、兴奋剂、类固醇使用建议

#### 处理方式

1. 服务层先做输入拦截
2. 输出生成后再做二次扫描
3. 统一返回安全拒答模板，不进入正常问答流程

## 5. Prompt 模板分层

AI 服务已按四层模板实现：

1. `system`：长期角色、安全边界、补剂原则、输出约束
2. `task`：当前场景目标与字段要求
3. `context`：用户档案、计划、检索片段、边界说明
4. `output`：固定 JSON 字段与数量限制

## 6. 输出校验策略

1. 优先要求 JSON 输出。
2. 服务层先做 `json.loads` 解析。
3. 解析失败时尝试截取首个 JSON 对象做一次自动修复。
4. 修复失败则返回结构化异常，后续可接静态兜底。
5. 所有输出进入安全扫描。

## 7. 语气与文风

1. 简洁直接
2. 不使用夸张营销话术
3. 不羞辱用户
4. 不制造焦虑
5. 不承诺短期神奇效果
6. 像执行型教练，不像鸡汤文案

## 8. 常见失败模式与约束

### 8.1 失败模式

1. 模型擅自改动热量和组次数值
2. 回答过于宽泛，没有可执行内容
3. 对补剂回答过度积极
4. 把普通问题错误上升为医疗建议
5. 明明是规则引擎改计划，却让模型重新计算结构化结果

### 8.2 对应约束

1. 在 Prompt 中明确“不得修改规则结果”
2. 要求输出 1-3 条具体行动建议
3. 固定补剂原则
4. 高风险问题触发拒答与转介模板
5. 一键改计划场景必须先有规则引擎结果，再允许模型解释

## 9. 规则引擎与 LLM 边界

### 9.1 规则引擎负责

1. 热量、宏量营养、训练组次数值的核心计算
2. 一键改计划的结构化改写与动作替换
3. 高风险输入识别与拒答
4. 模型输出异常时的静态兜底

### 9.2 LLM 负责

1. 饮食计划解释
2. 训练计划解释
3. 改计划后的自然语言说明
4. 基于检索片段的基础问答组织

### 9.3 协作顺序

1. 先做安全检查
2. 再做知识检索
3. 需要改计划时由规则引擎先输出结构化结果
4. 最后由 LLM 解释，并做输出安全扫描

## 10. 示例拒答模板

当问题涉及医疗、极端减重或高风险内容时，统一输出：

```text
这个问题超出 CampusFit AI 当前可安全提供的范围。我可以继续帮你调整饮食和训练执行方案，但涉及医疗或高风险情况，建议及时咨询医生或持证专业人士。
```
