# 周训练模板与休息规则设计

## 目标

在保留现有通用训练模板体系的前提下，新增一套按周配置的训练模板能力，让后台可以按周一到周天维护训练计划，并把“复合动作 3-4 分钟、孤立动作 2-3 分钟”的休息判定规则落到训练方案生成和用户端展示里。

## 现状

1. 后台训练模板仍是单层 `items` 列表，不支持按周配置。
2. 规则引擎只会按经验等级和训练日数量输出通用模板，不支持用户自定义的固定周计划。
3. 用户端训练方案只展示动作、组次、休息秒数，不展示动作类型，也无法解释休息时长来源。

## 设计结论

### 1. 保留旧体系，新增周模板扩展层

- 保留现有 `TrainingTemplate` 和通用规则引擎逻辑，避免影响现有训练生成链路。
- 在后台新增“周训练模板”模型，用于描述一周七天的训练安排。
- 第一版把“超级组”“递减组”继续放在动作备注里，不做结构化建模。

### 2. 周模板数据结构

周模板包含两层：

1. 模板头信息
- 模板名称
- 目标类型
- 经验等级
- 每周训练天数
- 状态
- 版本
- 强度等级
- 标签
- 说明

2. 周配置 `weekDays`
- 固定七天：周一到周天
- 每天字段：
  - `weekday`
  - `dayType`：`training` 或 `rest`
  - `title`
  - `notes`
  - `items`

训练动作字段：

- `exerciseCode`
- `exerciseName`
- `sets`
- `reps`
- `movementPattern`：`compound` 或 `isolation`
- `restSeconds`
- `restRuleSource`：`system` 或 `manual`
- `notes`
- `displayOrder`

### 3. 休息判定规则

规则分两层：

1. 区间规则
- 复合动作：180-240 秒
- 孤立动作：120-180 秒

2. 系统默认值
- 复合动作默认写入 `210`
- 孤立动作默认写入 `150`

这样既保留了你给的区间规则，又让系统有确定秒数可用于展示、计时和持久化。

### 4. 动作识别策略

第一版采用“白名单 + 名称关键词”方式识别动作类型：

- 先匹配 `exerciseCode`
- 再匹配动作名称关键词
- 匹配不到时默认按 `isolation`

原因：

- 这套规则稳定、可测试、可解释
- 避免一开始就用模糊推断把休息时间算错
- 后续如果要提高精度，再加后台手动修正即可

### 5. 后台改造方向

- 保留现有训练模板列表与详情页。
- 新增“周训练模板”列表、详情和编辑表单。
- 编辑器采用“模板头信息 + 七天卡片”的布局。
- 休息日隐藏动作列表，仅保留恢复建议。
- 动作输入时自动根据动作名填充 `movementPattern` 和默认 `restSeconds`。
- 如果管理员手动修改休息秒数，记录为 `manual`，后续不再自动覆盖。

### 6. 用户端改造方向

Web 和小程序训练方案都补充以下信息：

- 动作类型标签：复合 / 孤立
- 休息来源：系统建议 / 手动设置
- 休息说明：
  - 复合动作显示“主项动作，建议更长恢复”
  - 孤立动作显示“辅助动作，恢复时间较短”

休息日不再显示空动作列表，而是明确显示“休息 / 轻活动日”和恢复建议。

### 7. 训练生成链路

- 通用规则引擎继续负责旧逻辑。
- 新增一套“原始周模板”默认模板数据，按你的周计划预置。
- 训练方案生成时优先尝试按周模板生成；没有命中周模板时，回退到旧规则引擎模板。

## 你的模板映射

- 周一：休息
- 周二：胸 / 肩 / 三头
- 周三：背 / 二头
- 周四：腿
- 周五：休息
- 周六：胸 / 肩 / 三头
- 周天：背 / 二头

动作类型按规则表自动识别，超级组和递减组写入动作备注。

## 风险与控制

1. 后台和 API 目前是两套训练模板来源
- 后台训练模板走本地 mock store
- 用户训练方案走 Nest API + rule-engine
- 本次会同时补齐两边结构，但不强行合并数据源

2. 现有 `TrainingPlanItem` 数据库表没有动作类型字段
- 本次先在训练生成结果和接口序列化里增加该信息
- Prisma 模型同步补字段，保证后续可持久化

3. 组合动作暂不结构化
- 先保证主链路可用
- 后续如需精确展示超级组和递减组，再单独升级动作模型
