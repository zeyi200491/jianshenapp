# 饮食计划替代录入设计

**日期：** 2026-04-16

**目标：** 在饮食计划页支持按饮食环境生成差异化推荐，并允许用户为早餐、午餐、晚餐录入实际摄入食物，系统按内置食物库和份量档位自动估算营养，并回写当天汇总。

## 一、背景与问题

当前系统已经在建档阶段采集 `dietScene`，并且规则引擎会按 `canteen / dorm / home` 生成基础饮食建议。但现状存在两个问题：

1. 产品层对饮食环境的表达不够贴近用户心智。
   - 用户真正关心的是“学校有什么吃什么”和“我能自己做饭”这两种模式。
   - 现在的 `宿舍` 和 `家庭` 在产品层差异不大，反而增加理解成本。

2. 饮食计划页只能展示推荐，不能承接真实执行。
   - 用户如果某餐没有按推荐吃，只能自己心算偏差。
   - 系统无法根据真实摄入重算热量、蛋白质、碳水、脂肪，也无法把真实摄入继续传给后续页面和能力。

## 二、产品目标

本次改造只解决两个核心问题：

1. 对外把饮食环境收敛为两类：
   - `食堂`
   - `可做饭`

2. 在饮食计划页支持三餐替代录入：
   - 用户选择一餐
   - 搜索并选择内置食物库中的常见食物
   - 选择 `小份 / 中份 / 大份`
   - 系统自动估算营养并覆盖该餐展示与当天营养汇总

## 三、范围

### 3.1 本次范围内

- 建档页和饮食计划页对外只显示 `食堂 / 可做饭`
- 规则引擎与页面文案根据两类场景输出不同推荐风格
- 三餐支持替代录入
- 替代录入的输入方式固定为：
  - `食物名`
  - `份量档位`
- 食物营养由后端内置食物库自动估算
- `/today` 聚合接口返回每餐的计划值、实际值、生效值和当天生效汇总
- 饮食计划页 `/status` 展示替代结果并支持恢复推荐

### 3.2 本次明确不做

- 多食物组合录入
- 用户自己手填营养值
- 图片识别
- 自然语言解析整句菜单
- 历史常吃收藏与复用
- 独立的饮食日志中心

## 四、核心设计

### 4.1 饮食环境收敛策略

为了避免立即做数据库迁移，本次采用“产品层收敛，底层兼容”的方式：

- `canteen -> 食堂`
- `dorm -> 可做饭`
- `home -> 可做饭`

短期内数据库和既有老数据仍保留原始枚举值，前后端在展示与规则输出时做统一映射。

### 4.2 推荐值与实际值分离

系统里的餐次信息分为三层：

- `planned`
  - 系统根据规则引擎生成的推荐餐
- `actual`
  - 用户实际录入的替代餐
- `effective`
  - 页面展示和汇总使用的生效餐
  - 有 `actual` 时取 `actual`
  - 否则取 `planned`

这样可以同时保留“系统建议吃什么”和“用户实际吃了什么”。

### 4.3 食物估算模型

食物估算采用内置食物库，不开放自由营养输入。

每个食物条目包含：

- `code`
- `name`
- `aliases`
- `sceneTags`
- `nutritionPerMedium`
  - `calories`
  - `proteinG`
  - `carbG`
  - `fatG`
- `portionMultipliers`
  - `small`
  - `medium`
  - `large`

约定：

- 中份作为基准份量
- 小份和大份通过倍率换算
- 返回值统一向整数取整

### 4.4 三餐替代录入规则

- 支持 `breakfast / lunch / dinner`
- 每餐同一时间只保留一条当前有效替代记录
- 用户再次保存同一餐时，直接覆盖旧记录
- 用户删除替代记录后，该餐恢复显示推荐值

## 五、数据结构设计

### 5.1 食物库

第一阶段不建数据库表，先以服务端常量方式维护，便于快速迭代和补充词条。

后续如果食物词条规模扩大，再独立为数据库表或配置中心。

### 5.2 实际摄入覆盖表

新增 `MealIntakeOverride`，用于持久化用户对某天某餐的真实摄入。

建议字段：

- `id`
- `userId`
- `dailyPlanId`
- `mealType`
- `source`
- `foodCode`
- `foodNameSnapshot`
- `portionSize`
- `calories`
- `proteinG`
- `carbG`
- `fatG`
- `createdAt`
- `updatedAt`

约束：

- `@@unique([dailyPlanId, mealType])`

设计理由：

- 推荐值存放在 `DietPlanItem`
- 实际值存放在 `MealIntakeOverride`
- 两者分离后，既能看偏差，也便于后续复盘和建议生成

## 六、接口设计

### 6.1 搜索食物

`GET /meal-foods/search?q=<keyword>&scene=<scene>`

用途：

- 返回内置食物库中与关键字匹配的常见食物
- 供前端搜索候选项

返回内容：

- 食物基础信息
- 中份营养
- 份量档位支持情况

### 6.2 保存餐次替代

`PUT /daily-plans/:dailyPlanId/meals/:mealType/intake`

请求体：

- `foodCode`
- `portionSize`

服务端职责：

- 校验当前用户是否拥有该 `dailyPlanId`
- 根据 `foodCode` 命中食物库
- 根据份量档位换算营养
- 对应餐次做 upsert
- 返回当前餐次 `actual` 和当天 `effectiveDailyTotals`

### 6.3 删除餐次替代

`DELETE /daily-plans/:dailyPlanId/meals/:mealType/intake`

服务端职责：

- 删除对应餐次的 override
- 返回恢复后的当天 `effectiveDailyTotals`

## 七、`/today` 聚合返回改造

`/today` 继续作为前端聚合入口，但结构扩展为：

- `weeklyDietPlan.displayScene`
- 每餐新增：
  - `planned`
  - `actual`
  - `effective`
- 新增：
  - `effectiveDailyTotals`

这样 `/status` 和其他后续页面都不需要自己拼接覆盖逻辑，统一消费聚合结果即可。

## 八、前端页面交互

### 8.1 饮食计划页 `/status`

每个餐次卡片新增：

- `录入实际吃了什么`
- `改成别的`
- `恢复推荐`

推荐交互：

- 在餐卡内展开轻量编辑区
- 输入关键词搜索食物
- 选择候选食物
- 选择 `小份 / 中份 / 大份`
- 预览估算营养
- 确认保存

### 8.2 展示规则

默认展示 `effective`

当餐次被替代时：

- 标题显示实际食物名和份量
- 打 `已按实际摄入重算` 标签
- 额外展示一行原推荐信息，保留对照关系

顶部汇总区使用 `effectiveDailyTotals`

如果实际摄入与推荐值有明显偏差，可显示一句偏差提醒。

## 九、错误处理

- 搜不到食物：
  - 提示用户换更常见的叫法
- 保存失败：
  - 不修改当前展示
  - 提示保存失败
- 恢复失败：
  - 保持当前状态
  - 提示稍后重试
- 鉴权失效：
  - 复用现有登录重定向

## 十、测试策略

### 10.1 后端单元测试

- 食物搜索匹配与别名匹配
- 小中大份倍率换算
- 同一餐重复提交覆盖旧记录
- 删除 override 后恢复推荐
- `effectiveDailyTotals` 汇总优先级正确

### 10.2 聚合服务测试

- 无 override 时返回原计划
- 单餐 override 只影响对应餐
- 三餐 override 时全天汇总全部按实际值
- `dorm/home` 对外展示统一为 `可做饭`

### 10.3 前端页面测试

- 场景文案展示正确
- 餐次替代后卡片和汇总同步更新
- 恢复推荐后回退正常
- 搜索空结果与接口异常提示正常

## 十一、验收标准

- 建档页和饮食计划页只展示 `食堂 / 可做饭`
- 食堂和可做饭的推荐风格有明显区别
- 三餐均可进行替代录入
- 替代录入仅依赖内置食物库和份量档位
- 保存后该餐与当天汇总会立即重算
- 删除后可恢复推荐值
- 不破坏现有今日页、打卡页和鉴权流程
