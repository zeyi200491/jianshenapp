# 用户信息架构重做设计

日期：2026-04-18

## 目标

重做网页端与用户信息相关的信息架构，解决当前“业务入口”和“开发入口”混杂的问题。

本次设计只覆盖以下目标：

- 保留 `/status` 路由，但在语义上彻底正名为“饮食计划页”
- 首页隐藏开发态服务信息，不再向普通用户展示运行状态
- 将“设置 / 帮助”从伪入口改成真实入口
- 新增“个人中心”页，统一承接设置、帮助和退出登录

本次设计明确不做：

- 不新增后端接口
- 不引入开发诊断入口
- 不把个人中心扩展成完整资料编辑系统
- 不修改 `/status` 的 URL

## 问题陈述

当前工作区存在三类信息架构问题：

1. 首页同时承担产品介绍和开发诊断展示，普通用户会看到“本地服务状态”“开发可见”等开发语义。
2. `/status` 已经在实现层面变成饮食计划页，但历史命名和上下文仍容易让人理解成系统状态页。
3. 登录后工作台中的“设置 / 帮助”并不是独立真实入口，而是跳向不相干页面，破坏了用户预期。

这些问题共同导致用户心智不稳定：用户无法判断哪里是“业务功能”，哪里是“开发工具”，也无法形成统一的个人入口认知。

## 设计原则

1. 业务优先：用户可见入口只表达业务语义，不暴露开发态概念。
2. 入口收口：与用户自身相关的信息统一收口到“个人中心”。
3. 渐进改造：优先复用现有数据与页面能力，不发明新后端能力。
4. 低风险回归：主导航骨架保持稳定，只重整错误入口和错误表达。

## 方案比较

### 方案 A：最小信息架构重整

做法：

- 保留工作台主导航：仪表盘、饮食计划、每日打卡、每周复盘
- 新增 `/account` 作为“个人中心”
- 首页删除运行状态展示
- 顶部齿轮和底部“设置 / 帮助”全部收口到 `/account`

优点：

- 改动集中
- 回归风险最低
- 能直接解决假入口和开发态暴露问题

缺点：

- 个人中心第一版功能相对克制

### 方案 B：中等重整

做法：

- 在方案 A 基础上，进一步重做首页角色和顶部交互区

优点：

- 体验更完整

缺点：

- 超出本次需求
- 需要更多 UI 和交互回归

### 方案 C：整套导航重构

做法：

- 重新划分公开页、工作台、账户体系的全部层级

优点：

- 长期结构最干净

缺点：

- 范围明显过大
- 不适合当前迭代

### 结论

采用方案 A。

理由：它以最小改动解决当前最明确的三个问题，同时不触碰后端、不打断现有主流程，也不会把工作扩展成一轮全站导航重构。

## 目标信息架构

### 公开页

- `/`
  - 角色：产品价值说明与主入口页
  - 不再承担开发诊断展示职责
- `/login`
- `/onboarding`

### 登录后工作台

- `/today`：仪表盘
- `/status`：饮食计划
- `/check-in`：每日打卡
- `/review`：每周复盘
- `/assistant`：AI 助手
- `/account`：个人中心

## 页面与入口设计

### 1. 首页

首页保留以下角色：

- 产品定位说明
- 核心功能概览
- 主路径说明
- 登录后主入口跳转

首页删除以下内容：

- `RuntimeStatus` 组件
- “本地服务状态”
- “开发可见”
- 任何 API / AI 健康检查说明

首页新增或保留的用户入口应为：

- 进入今日仪表盘
- 邮箱登录
- 个人中心

### 2. `/status` 页面

保留 URL `/status`，但只按“饮食计划”语义表达。

要求：

- 页面标题、导航名、模块标题、说明文字统一使用“饮食计划”语义
- 不出现“状态页”“服务状态”“运行状态”等系统诊断表达
- 保持其作为周菜单、餐次详情、替换方案、执行指导页面的业务定位

### 3. 个人中心 `/account`

个人中心采用单页分区结构，不拆成多路由。

#### 账号信息区

- 显示昵称、登录状态、建档状态
- 提供前往建档或回到今日页的动作

#### 我的设置区

- 显示当前已有的配置摘要，如训练目标、饮食场景、训练频率
- 第一版只做“查看当前配置 + 跳转到对应页面调整”
- 不新增保存接口，不在本次迭代中扩展为完整设置表单

#### 帮助支持区

- 展示 FAQ
- FAQ 内容围绕真实业务流程：
  - 如何开始使用
  - 饮食计划怎么看
  - 每日打卡后数据去哪了
  - AI 助手能做什么
- 不包含开发环境诊断信息

#### 账户动作区

- 退出登录
- 返回首页
- 返回今日页

### 4. 工作台导航与顶部入口

#### 侧边栏

- 保留主导航四项：仪表盘、饮食计划、每日打卡、每周复盘
- 底部不再分别放“设置 / 帮助”两个伪入口
- 改为一个真实的“个人中心”入口

#### 顶部区域

- 齿轮按钮跳转到 `/account`
- AI 助手入口保持独立
- 铃铛按钮如果继续保留，只作为视觉元素；不在本次迭代中扩展通知系统

### 5. 公开页头部导航

- 增加“个人中心”入口
- 保持与登录后工作台的用户语言一致
- 不出现开发诊断或内部工具词汇

## 数据与实现策略

本次设计坚持复用已有能力：

- 账户信息优先复用本地 session 与 `fetchCurrentUser`
- “我的设置”优先复用已有 profile 数据
- 饮食计划页继续复用 `fetchToday` 及当前饮食替换能力

不新增：

- 新后端接口
- 新设置模型
- 新帮助系统服务

## 错误处理

所有新入口与现有页面保持一致的认证与异常处理：

- 未登录时跳转 `/login`
- 遇到 `401` 时清理本地 session 并跳转 `/login`
- 遇到 `CONFLICT` 时通过 `setStoredSessionOnboardingStatus(false)` 后跳转 `/onboarding`

## 验收标准

### 首页

- 不再出现 `RuntimeStatus`
- 不再出现“本地服务状态”
- 不再出现“开发可见”

### 饮食计划页

- `/status` 保持可访问
- 页面语义只表达“饮食计划”
- 不残留系统状态表达

### 个人中心

- `/account` 可访问
- 页面不是空壳，至少包含：
  - 账号信息
  - 我的设置
  - 帮助支持
  - 退出登录

### 入口一致性

- 工作台齿轮入口跳转到 `/account`
- 侧边栏底部入口跳转到 `/account`
- 公开页头部存在“个人中心”入口

### 验证命令

- `node .\apps\web\tests\smoke.test.mjs`
- `npm.cmd --prefix .\apps\web run build`

## 风险与约束

1. 当前部分源码文件存在字符编码历史问题，改动时需要避免把乱码扩散到新文件。
2. 个人中心第一版如果承担过多编辑能力，会快速膨胀为新需求集合，因此必须保持“查看 + 跳转”的克制边界。
3. 首页移除开发态状态后，开发者将失去该页面上的即时诊断入口，但这是本次设计明确接受的取舍。

## 实施建议

建议按以下顺序实施：

1. 先新增 `/account`，确保真实入口成立。
2. 再替换工作台与公开页中的“设置 / 帮助”入口。
3. 再清理首页开发态信息。
4. 最后更新 smoke 测试与构建验证。
