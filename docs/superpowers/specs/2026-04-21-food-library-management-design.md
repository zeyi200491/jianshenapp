# 2026-04-21 食物库管理与餐食替换搜索设计

## 1. 背景与问题

当前“搜索并替换今天这餐”功能并不查询数据库，也不是外部食物库搜索，而是直接读取后端代码中的内置常量 `FOOD_LIBRARY`。这带来三个直接问题：

1. 可搜索食物数量极少，很多常见替代食物根本不存在。
2. 搜索结果受场景过滤影响明显，用户会误以为“搜索坏了”。
3. 新增或修改食物必须改代码，无法通过后台维护。

现状证据：

- Web 搜索走 `/meal-foods/search`。
- API 搜索直接从 `FOOD_LIBRARY` 常量做名称/别名包含匹配。
- Admin 现有内容管理仍以 `apps/admin/app/api/v1/admin/*` 的本地 mock 路由为主，还没有真正落到 `apps/api` 数据库后端。

本次目标不是继续扩写死常量，而是一步到位建立“真实数据库食物库 + 后台管理 + Web 搜索替换”闭环。

## 2. 目标

### 2.1 功能目标

1. 将食物库从代码常量迁移为数据库实体。
2. Web 端搜索改为查询数据库食物库。
3. Admin 提供食物库管理能力，支持列表、筛选、新建、编辑、停用。
4. 用户选择候选食物后，仍然把“今天这餐的实际摄入”保存为快照，不受后续食物库修改影响。

### 2.2 非目标

本次明确不做以下内容：

1. 不接入外部营养数据库。
2. 不做 AI 语义搜索、拼音搜索、模糊纠错。
3. 不做复杂的营养相似度推荐算法。
4. 不做多版本营养值审计和发布流。

## 3. 方案对比与选择

### 3.1 方案 A：继续使用代码常量，只扩充条目

- 优点：开发最快。
- 缺点：核心问题未解决，依然不能后台维护。

### 3.2 方案 B：数据库食物库 + 真实后端 API + Admin 管理页

- 优点：一次解决“搜不到”和“无法维护”两个核心问题。
- 优点：改动集中，复杂度可控，适合当前仓库阶段。
- 缺点：需要同时改 Prisma、API、Admin、测试。

### 3.3 方案 C：高度规范化多表模型

- 优点：长期扩展性最好。
- 缺点：当前业务规模下明显过重，交付速度和维护成本都不划算。

### 3.4 结论

采用方案 B。

原因：当前最紧迫的问题是让食物库可维护、可搜索、可真正落地到用户替换流程，而不是先做一套学术上最完整的数据模型。

## 4. 设计原则

1. 先解决真实业务闭环，再做高级搜索。
2. 历史摄入记录必须保存快照，不与食物主数据强耦合。
3. 后台操作优先“停用”而不是“删除”，避免历史引用失真。
4. 优先复用当前前端交互和 API 结构，降低影响面。
5. 当前 Admin 仍是 mock 架构，本次以“新增真实后端链路”的方式落地，不顺手重构全部后台。

## 5. 数据模型设计

## 5.1 新增表：`FoodLibraryItem`

建议新增 Prisma 模型 `FoodLibraryItem`，字段如下：

- `id: String`
- `code: String`
- `name: String`
- `aliases: Json`
- `sceneTags: Json`
- `suggestedMealTypes: Json`
- `calories: Int`
- `proteinG: Int`
- `carbG: Int`
- `fatG: Int`
- `status: String`
- `sortOrder: Int`
- `createdAt: DateTime`
- `updatedAt: DateTime`

字段说明：

- `code`：稳定业务标识，用于保存用户替换记录。
- `aliases`：先以字符串数组 JSON 存储，降低实现成本。
- `sceneTags`：存储 `canteen`、`cookable` 等可用场景。
- `suggestedMealTypes`：存储 `breakfast`、`lunch`、`dinner`。
- `status`：至少支持 `active`、`inactive`。
- `sortOrder`：用于后台人工调序与搜索结果兜底排序。

## 5.2 不修改表：`MealIntakeOverride`

`MealIntakeOverride` 继续保存：

- `foodCode`
- `foodNameSnapshot`
- `portionSize`
- 四项营养值快照

原因：

1. 用户当日摄入是业务快照，不应该随食物库编辑而改变。
2. 这也允许管理员后续修正食物库营养值，而不污染历史记录。

## 5.3 初始化数据

把现有 `FOOD_LIBRARY` 中的内置 8 条食物迁移为数据库种子数据，作为第一批正式数据。

## 6. 后端设计

## 6.1 模块拆分

在 `apps/api/src/modules/meal-intakes` 中完成本次主体改造，保留现有替换写回逻辑，并将数据来源由常量切换为数据库。

建议新增：

- `food-library.repository.ts`
- `food-library.service.ts`

也可以先不新开模块，仍放在 `meal-intakes` 模块内部，避免横向扩散过大。

## 6.2 搜索接口

保留现有接口：

- `GET /meal-foods/search?q=xxx&scene=xxx`

实现变化：

1. 从数据库查询 `status = active` 的食物。
2. 若有 `scene`，只返回场景命中的食物。
3. 支持名称和别名的包含匹配。
4. 返回结构保持兼容 `MealFoodSearchResult`，避免前端改动过大。

排序规则：

1. 名称完全等于关键词优先。
2. 名称包含关键词优先于别名包含关键词。
3. `sortOrder` 升序。
4. `updatedAt` 倒序作为次级稳定排序。

说明：本次不引入全文索引和复杂评分，先保证结果可控、可解释。

## 6.3 替换写入接口

保留现有接口：

- `PUT /daily-plans/:dailyPlanId/meals/:mealType/intake`

实现变化：

1. `foodCode` 从数据库表查找，而不是从常量查找。
2. 读取当前食物的营养值，结合 `portionSize` 计算实际写入值。
3. 继续写入 `MealIntakeOverride` 快照字段。

## 6.4 后台管理接口

在 `apps/api` 新增真实管理接口：

- `GET /admin/food-library-items`
- `GET /admin/food-library-items/:id`
- `POST /admin/food-library-items`
- `PATCH /admin/food-library-items/:id`

本次不提供物理删除接口，停用通过更新 `status = inactive` 完成。

列表接口支持过滤：

- 关键词
- 状态
- 场景

## 6.5 鉴权策略

如果当前 `apps/api` 还没有统一后台管理员鉴权，本次先按项目现状做最小实现：

1. 先把接口划到 `admin` 命名空间。
2. 认证与权限可先复用当前受保护接口机制。
3. 若管理员身份体系尚未完成，先在设计实现中显式标记为受限内部管理接口。

这部分不应阻塞食物库闭环，但必须在代码里留清晰边界。

## 7. Admin 设计

## 7.1 现实约束

`apps/admin` 当前大量功能仍通过本地 `app/api/v1/admin/*` 路由 + `mock-store` 提供数据。这意味着如果继续沿用 mock-store，新建“食物库管理”页面也只会改到本地假数据，无法真正影响 Web 搜索。

因此，本次食物库管理必须接入真实后端。

## 7.2 推荐接法

保持 Admin 前端调用方式不变，仍调用：

- `/api/v1/admin/food-library-items`

但在 `apps/admin/app/api/v1/admin/food-library-items/*` 中不再读 `mock-store`，而是做一层到 `apps/api` 的代理转发。

好处：

1. 不用一次重写整套 Admin 数据获取方式。
2. 新功能可先落到真实后端。
3. 对页面层和现有 `useApiResource`、`apiRequest` 改动最小。

## 7.3 后台页面

新增导航项：

- `食物库管理`

新增页面：

1. 列表页 `/food-library-items`
2. 新建页 `/food-library-items/new`
3. 详情页 `/food-library-items/[id]`
4. 编辑页 `/food-library-items/[id]/edit`

## 7.4 列表页能力

列表页应支持：

1. 关键词搜索
2. 按状态筛选
3. 按场景筛选
4. 显示主要营养值
5. 显示推荐餐次与场景标签
6. 快速进入编辑

## 7.5 表单字段

表单应包含：

1. 食物名称
2. 食物编码 `code`
3. 别名
4. 场景标签
5. 推荐餐次
6. 热量
7. 蛋白质
8. 碳水
9. 脂肪
10. 状态
11. 排序值

交互建议：

- `aliases` 使用多行文本，一行一个别名。
- `sceneTags`、`suggestedMealTypes` 使用多选。
- 数值字段做非负整数校验。

## 8. Web 端行为变化

Web 页面整体交互不重做，保留现在的“搜索并替换今天这餐”流程，只替换数据源与提示语义。

建议小幅优化文案：

1. 把“从餐食库中匹配”改成更直白的“从可用食物库中搜索”。
2. 无结果时提示“当前场景下没有匹配项，可尝试更通用的食物名或联系后台补充食物库”。

这样用户能更快理解这不是全网搜索。

## 9. 迁移与兼容

## 9.1 数据迁移

新增 Prisma migration 创建 `food_library_items` 表。

## 9.2 种子迁移

新增 seed/初始化脚本，将旧 `FOOD_LIBRARY` 写入数据库。

要求：

1. 重复执行时不重复插入。
2. 以 `code` 作为幂等键。

## 9.3 兼容策略

正式切换时删除对硬编码常量搜索的依赖，但可以短暂保留一个内部转换函数用于导种，避免一次改动过大。

## 10. 测试设计

## 10.1 API 测试

新增或调整测试覆盖：

1. 名称命中搜索
2. 别名命中搜索
3. 场景过滤
4. 停用状态不返回
5. 根据 `foodCode` 成功写入餐次实际摄入
6. 无效 `foodCode` 返回 `NOT_FOUND`

## 10.2 Admin 测试

至少覆盖：

1. 表单校验
2. 列表筛选参数拼接
3. 新建与编辑成功流程

## 10.3 Web 测试

至少覆盖：

1. 搜索接口返回结果后正确展示
2. 无结果状态展示
3. 选择候选食物后刷新今日餐次显示

## 11. 风险与应对

### 风险 1：Admin 仍基于 mock 架构，真实后端接入成本被低估

应对：

本次只把“食物库管理”作为第一条真实后端链路，不顺带重构全部 Admin。

### 风险 2：旧常量迁移后，搜索结果数量仍然太少

应对：

把导入旧数据作为起点，同时支持后台持续录入更多食物。

### 风险 3：历史记录受食物编辑影响

应对：

坚持快照写入，不直接关联实时营养值。

### 风险 4：搜索结果体验仍然不够智能

应对：

本次先把可维护性和可搜性打通，后续再在数据库食物库之上叠加排序和推荐能力。

## 12. 实施拆分建议

建议按以下顺序实施：

1. Prisma 模型与 migration
2. 种子数据导入
3. API 搜索与替换逻辑切换到数据库
4. Admin 代理路由
5. Admin 列表页与表单页
6. Web 文案与状态细化
7. 回归测试

## 13. 验收标准

满足以下条件即认为本次完成：

1. 后台可新增、编辑、停用食物。
2. Web 搜索能查到后台新增食物。
3. 当前场景下只返回可用食物。
4. 替换后“今天这餐”的实际摄入正确更新。
5. 后续修改食物库不会篡改历史已保存摄入。
