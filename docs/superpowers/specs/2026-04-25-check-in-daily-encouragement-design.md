# 2026-04-25 每日打卡鼓励语按日期轮换设计

## 1. 背景与问题

当前前端里有两处与你这次需求直接相关的内容：

1. `apps/web/app/status/page.tsx` 的饮食计划页头部，存在一段解释性文案：说明当前页面只展示 `today` 接口返回的真实周菜单、今日餐次和实际摄入状态。
2. `apps/web/app/check-in/page.tsx` 的每日打卡页头部，存在一条写死的固定励志语录。

这两个实现都不再匹配当前目标：

- 饮食计划页需要把这段说明文案去掉，不再占据页面头部位置。
- 每日打卡页需要把固定语录改成“按日期固定轮换”的鼓励语，即同一个 `YYYY-MM-DD` 对所有用户显示同一句，第二天自动切换到下一句。

## 2. 目标

### 2.1 功能目标

1. 删除饮食计划页头部现有的说明文案，不追加新的占位文案。
2. 在每日打卡页头部展示每日鼓励语，替换现有固定语录。
3. 鼓励语按日期固定轮换，同一天所有用户看到相同内容。
4. 用户切换每日打卡页日期时，鼓励语同步切换到对应日期的文案。

### 2.2 非目标

本次不做以下内容：

1. 不改后端接口，不新增接口字段。
2. 不做后台配置鼓励语，不做运营位管理。
3. 不引入用户个性化随机文案。
4. 不改每日打卡页整体布局和视觉结构。

## 3. 方案对比与选择

### 3.1 方案 A：前端本地文案池 + 按日期取模

- 做法：在 Web 端维护一组鼓励语，根据日期计算稳定索引并返回文案。
- 优点：改动最小；无需后端参与；行为稳定；单元测试简单。
- 缺点：文案内容写死在前端，后续改文案需要发版。

### 3.2 方案 B：前端维护全年映射表

- 做法：预先给 365 天甚至 366 天都分配固定文案。
- 优点：可以做到全年不重复，节奏完全可控。
- 缺点：维护成本过高；对当前需求明显过重。

### 3.3 方案 C：后端返回每日鼓励语

- 做法：由接口直接返回当天文案，前端只负责展示。
- 优点：后续最适合做运营配置、节日文案和多端统一。
- 缺点：需要改接口、联调和测试链路；对本次小需求成本偏高。

### 3.4 结论

采用方案 A。

原因很直接：这次只是一个前端文案位改造，目标是快速稳定上线，不值得把后端链路一起带进来。按日期固定轮换也非常适合在前端通过纯函数完成。

## 4. 设计原则

1. 文案选择逻辑必须稳定，同一日期在任何客户端都得到相同结果。
2. 优先复用现有展示辅助层，把新逻辑放入已有的 `apps/web/lib/check-in-view.ts`。
3. 页面组件只负责取值和展示，不在页面里硬编码轮换算法。
4. 先补测试，再做页面替换，保证日期轮换行为可验证。

## 5. 详细设计

### 5.1 饮食计划页改动

实际路由 `/diet` 当前由 `apps/web/app/diet/page.tsx` 直接导出 `../status/page`，因此真正需要改的是：

- `apps/web/app/status/page.tsx`

改动内容：

1. 删除页头 `h1` 下方那段说明性段落。
2. 保留标题、指标卡和页面结构不变。

预期结果：

- 页面头部更干净；
- 不再占用用户注意力解释“真实数据”来源；
- 不引入新的空白区块或额外占位文案。

### 5.2 每日打卡页改动

改动文件：

- `apps/web/app/check-in/page.tsx`
- `apps/web/lib/check-in-view.ts`

改动方式：

1. 在 `check-in-view.ts` 新增本地鼓励语文案池常量。
2. 新增纯函数，根据传入的日期字符串返回稳定文案。
3. `check-in/page.tsx` 不再写死励志语录，而是调用该纯函数并展示结果。

### 5.3 日期轮换规则

输入：

- 使用每日打卡页当前的 `date` 值，格式为 `YYYY-MM-DD`。

规则：

1. 不直接使用浏览器当前时间做计算。
2. 以页面当前 `date` 为唯一输入。
3. 将日期转成稳定的数字，再对文案池长度取模，得到索引。
4. 结果只依赖日期字符串，因此同一天对所有用户一致。

推荐实现方式：

- 直接移除日期中的连字符，把 `YYYYMMDD` 转为数字；
- 使用 `Number(date.replaceAll('-', '')) % encouragements.length`；
- 返回对应文案。

这样做的原因：

- 可读性高；
- 不依赖时区推断；
- 对现有 `date` 参数格式足够稳定；
- 易于写单元测试。

### 5.4 文案池约束

文案池建议控制在 7 到 14 条之间，先满足“每天不同感受”的体验，再避免一次性塞入过多低质量文案。

文案要求：

1. 中文表达自然，不写鸡汤腔过重的长句。
2. 语气偏陪伴和鼓劲，长度控制在一到两句内。
3. 避免过度承诺、羞辱式表达或空泛口号。
4. 文案长度需适配当前头部区域，避免在移动端挤压布局。

## 6. 组件与数据流

### 6.1 组件职责

- `apps/web/lib/check-in-view.ts`
  - 负责鼓励语文案池定义
  - 负责日期到文案的映射函数
  - 负责被测试直接覆盖

- `apps/web/app/check-in/page.tsx`
  - 负责调用鼓励语函数
  - 负责在头部渲染文案

- `apps/web/app/status/page.tsx`
  - 负责移除不再需要的头部说明文案

### 6.2 数据流

1. 用户进入每日打卡页。
2. 页面从 URL 状态或默认逻辑得到 `date`。
3. 页面把 `date` 传给鼓励语函数。
4. 纯函数返回对应鼓励语。
5. 页面在头部渲染结果。
6. 用户切换日期时，`date` 更新，文案同步更新。

## 7. 错误处理与边界

### 7.1 日期格式边界

正常情况下，当前页已有 `date` 状态管理，输入应始终是合法的 `YYYY-MM-DD`。本次不额外做复杂容错，但建议鼓励语函数在异常输入下返回文案池第一条，避免页面出现空白。

### 7.2 文案池为空

本次实现中不允许文案池为空。若后续有人误删导致为空，测试应第一时间暴露问题。

### 7.3 布局边界

替换后的鼓励语仍使用当前页面已有的段落样式，不新增卡片、不改字号体系。如果文案池中个别句子过长，应在文案层面控制，而不是让布局承担风险。

## 8. 测试设计

测试文件：

- `apps/web/tests/view-helpers.test.mjs`

至少新增以下用例：

1. 同一日期多次调用，返回同一句文案。
2. 两个不同日期返回不同文案。
3. 日期轮换基于传入的页面日期，而不是系统当前时间。
4. 异常输入时返回兜底文案（如果实现该兜底分支）。

饮食页文案删除本身不一定需要新增单测；若当前仓库没有页面级渲染测试，可通过现有类型检查和手工验收完成确认。

## 9. 验收标准

满足以下条件即可认为本次完成：

1. `/diet` 页头不再显示“这一页只展示 today 接口已经返回的真实周菜单...”这段文案。
2. `/check-in` 页头不再显示固定名言。
3. `/check-in` 页头显示按日期固定轮换的鼓励语。
4. 同一个日期所有用户看到的文案一致。
5. 切换每日打卡日期时，文案会跟着切换。
6. 相关辅助函数测试通过。

1. 在 `view-helpers.test.mjs` 先补鼓励语轮换测试，确认红灯。

4. 在 `status/page.tsx` 删除饮食页头部说明文案。
5. 运行 Web 相关测试，确认绿灯。
