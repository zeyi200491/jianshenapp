# 训练模板文字导入设计

## 目标

在现有“个人周训练模板”能力上，新增“粘贴自然语言训练文本并导入模板”的功能，让用户可以：

1. 在模板编辑器中粘贴一段中文训练计划文本。
2. 先查看解析预览，再确认是否覆盖模板。
3. 只更新本次文本里识别成功且用户勾选确认的周几。
4. 对识别失败的行看到明确报错，并回到原文修正后重新解析。
5. 在标准动作可匹配时自动标准化，匹配不上时按自由文本动作保存。

本次同时修复训练模板相关页面和接口摘要中的中文乱码，避免新功能落地后继续暴露异常文案。

## 非目标

本次不做以下内容：

1. 不引入 AI 解析训练文本，第一版只做规则解析。
2. 不做“预览区逐行动作表格编辑器”，用户修正错误时仍回到原始文本修改后重新解析。
3. 不要求用户一次输入完整 7 天，允许只导入部分天。
4. 不新增独立的导入历史管理页。
5. 不重构现有“模板应用到今天”的主链路，只保证导入后的模板能继续被该链路消费。

## 现状与问题

当前项目已经具备以下能力：

1. 用户可在 `/account/training-templates` 维护个人周模板。
2. today 页可预览模板并“应用到今天”，生成当日训练覆盖快照。
3. 模板项当前以结构化动作字段为主，适合标准动作录入，不适合直接承载复杂自然语言表达。

当前缺口有两类：

1. 用户从教练、笔记、聊天记录复制来的周计划文本，不能一键导入，只能手工逐项录入。
2. 训练模板相关页面和接口摘要存在中文乱码，影响当前功能可用性和信任感。

## 设计结论

### 1. 采用“后端解析预览 + 前端确认覆盖”

本次不在前端本地维护一套解析规则，而是新增后端解析接口：

1. 前端只负责采集原始文本、展示预览、提交确认。
2. 后端统一负责周几识别、动作行拆分、标准动作匹配、错误与警告生成。
3. 这样规则只维护一份，后续 Web、后台、移动端都能复用。

### 2. 导入作为模板编辑器的辅助入口

“文字导入”不新开独立业务模块，直接放进现有模板编辑器：

1. 用户在模板编辑页点击“文字导入”。
2. 在抽屉中粘贴文本并点击“开始解析”。
3. 查看按周几分组的预览、错误、警告和匹配状态。
4. 勾选本次要覆盖的周几并确认。
5. 系统只更新这些被确认的天，其余天保持不变。

### 3. 容错策略采用“部分成功、部分阻塞”

解析结果按行区分：

1. 成功：可直接导入。
2. 警告：允许导入，但要显式提示，例如未匹配标准动作、复合次数只能按文本保存。
3. 阻塞错误：该行所在周几默认不可确认覆盖，直到用户修正原文后重新解析。

### 4. 动作落库采用“双轨”

动作优先标准化，但允许自由文本兜底：

1. 能匹配标准动作库时，保留 `exerciseCode` 与标准展示名。
2. 匹配不上时，按自由文本动作保存，不强制用户补动作库。
3. 为了支持 `8+10+15×2`、`超级组`、组合动作等写法，需要扩展模板项字段。

## 用户流程

### 1. 导入入口

在训练模板编辑页顶部操作区新增：

1. `文字导入` 按钮。
2. `示例格式` 入口。
3. 一句短提示：`支持粘贴周计划文本，先预览再覆盖对应天`。

### 2. 导入流程

1. 用户打开模板编辑页。
2. 点击 `文字导入`。
3. 在抽屉中粘贴原始训练文本。
4. 点击 `开始解析`。
5. 查看解析结果：
   - 识别到的周几。
   - 每天的标题、休息日状态、动作列表。
   - 每个动作的匹配状态。
   - 逐条错误和警告。
6. 勾选本次要覆盖的周几。
7. 点击 `确认覆盖`。
8. 抽屉关闭，模板编辑器显示最新结果，并给出“已更新 N 天”的反馈。

### 3. 覆盖规则

1. 只覆盖本次识别成功且用户勾选的周几。
2. 未出现在文本中的周几保持原样。
3. 被识别为 `休息` 的周几，会清空该天动作并标记为休息日。
4. 有阻塞错误的周几默认不可勾选。

## 解析规则

第一版采用“规则解析 + 原文兜底 + 显式报错”。

### 1. 周几标题识别

支持以下常见写法：

1. `周一 休息`
2. `周二 胸 肩 三头`
3. `星期三 背 二头`
4. `周日 背二头`
5. `周天 背 二头`

规则：

1. 行首先识别周几。
2. 周几后的剩余文本作为当天标题。
3. 标题包含 `休息` 时，该天标记为 `rest`。
4. 其余情况默认标记为 `training`。

### 2. 动作行识别

优先支持这些模式：

1. `动作名 8×4`
2. `动作名 8x4`
3. `动作名 8✖️4`
4. `动作名 10*3`
5. `动作名 8+10+15×2`
6. `动作名 10+10×3`
7. `动作名 8×4（备注）`
8. `动作名 8×4(备注)`

解析结果尽量拆成：

1. `exerciseName`
2. `sets`
3. `reps`
4. `repText`
5. `notes`
6. `rawInput`

说明：

1. 纯数字次数可同时写入 `reps` 和 `repText`。
2. 复合次数如 `8+10+15` 只保证写入 `repText`，`reps` 可为空。
3. 括号内容优先进入 `notes`。

### 3. 复杂动作写法

对这类输入：

1. `二头超级组（站姿+坐姿 10+10×3）`
2. `推举➕腿弯曲 10+15×3`

第一版策略：

1. 尽量识别动作主名和组数。
2. 无法稳定拆分的重复次数、超级组说明、组合细节放进 `repText` 或 `notes`。
3. 如果仍无法可靠结构化，则保留整行 `rawInput` 并标记警告或错误。

### 4. 阻塞错误

以下情况视为阻塞错误：

1. 行无法归属到任何周几。
2. 行中既识别不到动作名，也无法作为自由文本动作保存。
3. 同一行结构冲突到无法判断组数与次数含义。
4. 重复周标题导致片段归属不清且无法安全合并。

### 5. 非阻塞警告

以下情况允许确认，但必须提示：

1. 未匹配到标准动作库。
2. 次数为复合写法，只能按 `repText` 保存。
3. 使用了特殊符号，系统按原文保留。
4. 动作名疑似重复，但不影响保存。

## 数据模型调整

本次不新建新的模板主表，沿用现有个人模板结构，只扩展模板项字段。

### 1. 模板项扩展

`user_training_template_items` 与对应的 override item 结构增加：

1. `rep_text`
   - 保存原始次数表达，如 `8+10+15`、`10+10`。
2. `source_type`
   - `standard | free_text`
3. `raw_input`
   - 保存原始动作行文本。

兼容规则：

1. 原有纯结构化动作继续正常工作。
2. today 页和“应用到今天”链路需要同步复制新字段。

### 2. 预览状态

解析预览不作为长期业务数据落库。第一版使用短时预览令牌：

1. 解析接口返回 `previewToken`。
2. 服务端用短时缓存或临时存储保存解析结果。
3. 确认覆盖时只提交 `previewToken + selectedWeekdays`。

这样可以避免前端篡改结构化结果，也避免专门维护长期导入历史。

## API 设计

### 1. 解析预览

新增：

`POST /users/me/training-templates/import-preview`

请求体：

```json
{
  "templateId": "uuid",
  "rawText": "周二 胸 肩 三头\n杠铃卧推 8×4"
}
```

返回体核心结构：

```json
{
  "previewToken": "token",
  "summary": {
    "detectedDays": 2,
    "successfulLines": 8,
    "warningLines": 2,
    "blockingLines": 1
  },
  "parsedDays": [
    {
      "weekday": "tuesday",
      "title": "胸 肩 三头",
      "dayType": "training",
      "selectable": true,
      "warnings": [],
      "items": [
        {
          "rawLine": "杠铃卧推 8×4",
          "exerciseName": "杠铃卧推",
          "matchedExerciseCode": "barbell-bench-press",
          "sets": 4,
          "reps": 8,
          "repText": "8",
          "notes": "",
          "matchStatus": "matched"
        }
      ]
    }
  ],
  "errors": [
    {
      "lineNumber": 12,
      "weekday": "sunday",
      "rawLine": "？？？",
      "message": "无法识别训练动作结构",
      "blocking": true
    }
  ]
}
```

### 2. 确认覆盖

新增：

`POST /users/me/training-templates/:id/import-apply`

请求体：

```json
{
  "previewToken": "token",
  "selectedWeekdays": ["tuesday", "wednesday", "saturday"]
}
```

行为：

1. 校验模板归属当前用户。
2. 校验 `previewToken` 仍有效。
3. 校验模板在预览后没有被其他操作改动到不可安全覆盖。
4. 只对用户勾选且无阻塞错误的周几执行覆盖。
5. 返回更新后的模板详情。

### 3. 不改动的接口

现有这些接口继续保留：

1. `GET /users/me/training-templates`
2. `GET /users/me/training-templates/:id`
3. `PATCH /users/me/training-templates/:id`
4. `POST /users/me/training-templates/:id/enable`
5. `POST /users/me/training-templates/:id/set-default`

导入能力是对现有模板管理接口的补充，而不是替代。

## 前端设计

### 1. 交互容器

使用抽屉而不是简单弹窗，原因：

1. 文本可能很长。
2. 预览结果按多天展示，内容高度不可控。
3. 错误与警告需要独立区域。

### 2. 抽屉结构

分为三块：

1. 输入区
   - 大文本框
   - `示例格式`
   - `开始解析`
2. 解析结果区
   - 按周几分组的卡片
   - 每天的标题、休息状态、动作列表、匹配状态
3. 问题区
   - 错误和警告分开展示
   - 错误项带行号和原文

### 3. 天卡片勾选

每个识别出的周几卡片都有勾选框：

1. 默认勾选所有无阻塞错误的周几。
2. 有阻塞错误的周几不可勾选。
3. 用户可取消部分天，实现局部覆盖。

### 4. 预览区不做逐行复杂编辑

第一版不在预览区内做完整动作编辑器：

1. 报错后提示用户回到原始文本修改。
2. 修改后重新点击 `开始解析`。

这样可以避免在抽屉里复制一套模板编辑器复杂逻辑。

### 5. 反馈与状态提示

确认覆盖后：

1. 关闭抽屉。
2. 模板编辑页高亮刚更新的周几。
3. 页面顶部显示：
   - `已更新 3 天训练模板`
   - `2 个动作按自由文本保存`

## 标准动作匹配

匹配策略分层执行：

1. 精确匹配标准动作名。
2. 常见别名或规范化名称匹配。
3. 失败后标记为 `free_text`。

不因为未匹配标准动作就阻止导入，除非整行动作结构本身不可保存。

## 乱码修复范围

本次将训练模板相关的用户可见乱码一并纳入范围，至少包括：

1. `apps/web/app/account/training-templates/page.tsx`
2. `apps/web/components/web/training-templates/training-template-editor.tsx`
3. `apps/web/components/web/training-templates/training-template-list.tsx`
4. `apps/web/components/web/today/training-plan-panel.tsx`
5. `apps/web/lib/use-today-dashboard.ts`
6. `apps/api/src/modules/training-templates/training-templates.controller.ts`

目标不是大规模文案重写，而是恢复正确中文并保证新导入能力的文案完整可读。

## 服务落点

### 1. `training-templates` 模块

负责：

1. 模板 CRUD。
2. 导入预览解析。
3. 导入确认覆盖。
4. 标准动作匹配与模板项写入。

### 2. `training-overrides` 模块

继续负责：

1. 把模板某一天应用到 today。
2. 生成和恢复当日 override。

新增字段要在这里同步复制，保证模板导入后的复杂动作信息不会在 today 链路里丢失。

### 3. `today.service.ts`

不新增导入职责，只需保证导入后的模板数据仍可被预览、应用、展示。

## 错误处理

### 1. 输入层

1. 空文本直接拦截。
2. 文本超过上限直接提示。
3. 未识别到任何周几，返回明确错误。

### 2. 解析层

每个错误至少返回：

1. 行号。
2. 原始行内容。
3. 归属周几或“未归属”。
4. 错误原因。
5. 是否阻塞确认。

### 3. 应用层

1. `previewToken` 失效时提示重新解析。
2. 模板在解析后被其他请求修改过时提示重新确认。
3. 只有勾选且无阻塞错误的周几会被更新。

## 测试策略

### 1. 后端解析单测

覆盖：

1. 休息日识别。
2. 周几标题多种写法。
3. `× / x / ✖️ / *` 乘号变体。
4. 括号备注。
5. 复合次数。
6. 超级组与组合动作。
7. 只贴部分天。
8. 异常行报错。

### 2. 后端接口测试

覆盖：

1. 预览成功。
2. 部分成功、部分失败。
3. 只覆盖勾选的部分天。
4. `previewToken` 失效。
5. 覆盖后模板内容正确。

### 3. 前端冒烟测试

覆盖：

1. 模板页存在 `文字导入` 入口。
2. 抽屉存在输入区、解析按钮、预览区、问题区。
3. 可勾选部分天确认覆盖。
4. 错误与警告文案可见。
5. 乱码文案已恢复可读。

### 4. 回归测试

覆盖：

1. 原有手工模板编辑不受影响。
2. today 预览模板与“应用到今天”链路不受影响。
3. 导入后生成的模板项在 today 与 override 中不丢字段。

## 风险与控制

### 1. 解析规则过松

风险：

为了追求“自由文本都能猜”，导致错误输入被误解析成错误动作。

控制：

1. 第一版坚持规则解析优先。
2. 拆不稳的内容宁可标警告或错误，也不静默乱猜。

### 2. 模板项字段兼容性

风险：

新增 `repText`、`sourceType`、`rawInput` 后，today 或 override 复制链路漏传字段。

控制：

1. 在模板项 presenter、override repository、today 聚合中统一补齐字段。
2. 用回归测试覆盖“导入后应用到今天”的完整链路。

### 3. 文案乱码继续扩散

风险：

只加新功能不修当前模板页乱码，会让用户对新能力产生不信任。

控制：

将乱码清理明确纳入本次范围，并加入前端冒烟检查。

## 推荐实施顺序

1. 先清点并修复训练模板相关页面与接口摘要里的乱码。
2. 扩展模板项与 override item 数据结构。
3. 新增导入预览与确认覆盖接口。
4. 在模板编辑器接入抽屉、预览、勾选确认。
5. 补后端解析测试、前端冒烟测试、today 回归测试。

这样可以先把现有模板体验恢复正常，再在稳定文案和结构上叠加导入能力，降低回归风险。
