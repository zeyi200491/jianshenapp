# CampusFit AI 用户故事

## 1. 用户角色

### 1.1 普通用户

主要通过网页完成建档、查看计划、打卡与复盘。

### 1.2 运营人员

主要使用后台维护模板、商品和内容，并监控关键数据。

## 2. 普通用户故事

### 2.1 登录与建档

1. 作为首次使用的用户，我希望通过邮箱验证码快速登录，这样我不需要复杂注册流程。
2. 作为目标明确的用户，我希望在建档时选择减脂、维持或增肌，这样后续计划能和我的目标匹配。
3. 作为食堂场景用户，我希望选择我的饮食场景是食堂，这样系统给出的计划更贴近日常生活。
4. 作为训练新手，我希望填写每周可训练天数和训练经验，这样系统不会给出超出能力的安排。
5. 作为对补剂不了解的用户，我希望能选择是否接受补剂建议，这样我能控制系统输出的内容范围。

### 2.2 今日页

1. 作为每天打开产品的用户，我希望第一眼看到今天要做什么，这样我不需要再去多个页面查找。
2. 作为执行力不足的用户，我希望今日页清楚显示饮食和训练计划是否已完成，这样我能快速知道今天还有哪些任务。
3. 作为碎片化使用的用户，我希望今日页能快速进入打卡、AI 助手和商品列表，这样流程更顺。

### 2.3 饮食计划

1. 作为不会算热量的用户，我希望系统直接给出今天吃什么和大概吃多少，这样我只需要照着执行。
2. 作为食堂场景用户，我希望每餐看到可替代的菜品选择，这样食堂没有某个菜时我也能执行。
3. 作为宿舍简做用户，我希望饮食建议尽量简单，这样我不需要复杂备餐。
4. 作为家庭烹饪用户，我希望知道每餐优先保证什么营养目标，这样我能灵活调整做法。

### 2.4 训练计划

1. 作为新手用户，我希望今天的训练动作不多且说明清晰，这样我更容易坚持。
2. 作为中级用户，我希望训练动作、组数和次数有明确要求，这样我能直接执行。
3. 作为时间紧张的用户，我希望在今天不方便训练时得到休息或轻活动替代建议，这样计划不会完全中断。

### 2.5 打卡与反馈

1. 作为容易放弃的用户，我希望打卡过程简单，这样我不会因为步骤太多而不记录。
2. 作为关注体重变化的用户，我希望可以顺手记录体重，这样系统能生成更准确的周复盘。
3. 作为复盘导向用户，我希望系统根据我的打卡情况总结本周表现，这样我知道问题出在哪里。

### 2.6 AI 助手

1. 作为新手用户，我希望问“今天为什么这样吃”，AI 能给我简单解释，这样我更愿意执行。
2. 作为遇到临时变化的用户，我希望问“今天没有鸡胸肉怎么办”，AI 能给出替代建议，这样计划更灵活。
3. 作为动作不熟悉的用户，我希望问“深蹲要注意什么”，AI 能给我基础注意事项，这样我更安全。
4. 作为担心补剂的用户，我希望问“乳清蛋白是不是必须”，AI 能明确说明补剂只是补充，这样我不会被误导。

### 2.7 商品浏览

1. 作为执行过程中需要补充器材或补剂的用户，我希望能看到基础商品列表，这样我不用跳到别的平台搜索。
2. 作为谨慎消费的用户，我希望商品展示包含适用场景和基础说明，这样我能判断是否值得看。

## 3. 运营人员故事

1. 作为运营人员，我希望能维护饮食模板和训练模板，这样不同类型用户都能获得基础可执行计划。
2. 作为运营人员，我希望能维护商品信息和标签，这样商品列表可以与用户需求挂钩。
3. 作为运营人员，我希望查看建档完成率、计划查看率和打卡率，这样我能判断 MVP 是否成立。

## 4. 用户故事验收关注点

1. 用户故事必须映射到明确页面与 API。
2. 每个故事都要能落到数据库实体和埋点事件。
3. 优先实现“可执行”和“可记录”，而不是“看起来很智能”。
